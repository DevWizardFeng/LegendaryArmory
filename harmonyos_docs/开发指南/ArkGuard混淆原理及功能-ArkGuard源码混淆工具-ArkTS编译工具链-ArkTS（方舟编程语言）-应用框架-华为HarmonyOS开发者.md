anchor-icon-link" anchorid="适用语言" tips="复制节点链接"></i></h3>          <p>ArkGuard支持ArkTS、TS和JS语言，不支持C/C++、JSON、资源文件等。</p>    </div>    <div class="tiledSection">     <h3 id="混淆能力">混淆能力<i class="anchor-icon anchor-icon-link" anchorid="混淆能力" tips="复制节点链接"></i></h3>          <p>ArkGuard支持名称混淆、代码压缩和注释删除的基础混淆功能，不支持控制流混淆、数据混淆等高级混淆功能。</p>     <p>名称混淆主要提供<strong>名称重命名</strong>和<strong>配置保留白名单</strong>的能力。</p>    </div>    <div class="tiledSection">     <h3 id="混淆能力局限性">混淆能力局限性<i class="anchor-icon anchor-icon-link" anchorid="混淆能力局限性" tips="复制节点链接"></i></h3>          <p><strong>1.语言的限制</strong></p>     <p>源码混淆工具在处理不同编程语言时，其类型分析机制、混淆策略和执行效率都会因目标语言的特性而呈现差异。以业界常用的ProGuard为例，其主要面向Java这类强类型语言进行混淆。由于强类型语言具有严格的类型系统，每个类型都有明确的定义来源。这种特性使得混淆过程中的类型关系追踪和处理更为精确，从而大幅减少了需要配置保留规则的场景。</p>     <p>相比之下，ArkGuard混淆工具主要针对JS、TS和ArkTS语言。JS支持运行时动态修改对象和函数，而混淆是在编译阶段进行的静态处理，这种差异可能导致混淆后的名称在运行时无法被正确解析，进而引发运行时异常。TS和ArkTS虽然引入了静态类型系统，但采用了结构性类型机制，即具有相同结构的不同命名类型会被视为等价类型。因此，在TS和ArkTS中仍然无法追溯类型的确切来源。</p>     <p>基于这些特性，使用ArkGuard时需要对更多语法场景进行白名单配置，同时，ArkGuard采用全局生效的属性保留机制，根据白名单统一保留所有同名属性，无法针对特定类型精确保留。</p>     <p>具体示例如下：</p>     <p>假设ArkGuard支持配置指定类型的白名单。配置类A1作为白名单，A1的属性prop1在白名单中，而A2的prop1属性不在白名单中。a2作为参数传入test函数，并在test函数内访问其属性。混淆前，可以正常访问prop1属性；混淆后，A1的属性prop1未被混淆，但A2的prop1属性被混淆，导致test函数中访问prop1属性时功能异常</p>     <p>因此，ArkGuard不支持针对特定类型的精确保留配置。</p>     <div _ngcontent-oqj-c106="" class="highlight-div"><div _ngcontent-oqj-c106="" class="highlight-div-header"><div _ngcontent-oqj-c106="" <p><strong>2.安全保证的有限性</strong></p>     <p>与其他源码混淆工具类似，混淆只能在一定程度上增加逆向工程的难度，并不能完全阻止逆向工程。</p>     <p>并且，由于ArkGuard混淆工具仅支持基础混淆功能，开发者不应只依赖ArkGuard来保证应用的安全性，对于源码安全有高要求的开发者，应考虑使用<a href="https://developer.huawei.com/consumer/cn/doc/harmonyos-guides/code-protect" target="_blank">应用加密</a>、安全加固等安全措施来保护代码。</p>    </div>    <div class="tiledSection">     <h2 id="混淆机制及流程">混淆机制及流程<i <p>在编译一个模块时，生效的混淆规则是<strong>当前编译模块混淆规则</strong>和<strong>依赖模块混淆规则</strong>的合并结果，具体规则如下：</p>     <p><strong>当前编译模块混淆规则</strong></p>     <p>指当前模块配置文件build-profile.json5中arkOptions.obfuscation.ruleOptions.files字段指定的混淆配置文件内容。</p>     <p><strong>依赖模块混淆规则</strong></p>     <p>根据依赖模块的类型，混淆规则分为以下两个来源：</p>     <ul>      <li>       <p><strong>本地HAR/HSP模块</strong></p>       <p>指该模块配置文件build-profile.json5中arkOptions.obfuscation.consumerFiles字段指定的混淆配置文件内容。</p></li>      <li>       <p><strong>远程HAR/HSP包</strong></p>       <p>指该远程HAR/HSP包中obfuscation.txt文件内容。</p></li>     </ul>     <p>构建HAP、HSP和HAR时，最终的混淆规则是以下文件的合并：</p>     <ul>      <li>当前构建模块的ruleOptions.files属性。</li>      <li>依赖的本地HSP的consumerFiles属性。</li>      <li>依赖的本地HAR的consumerFiles属性。</li>      <li>依赖的远程HAR和远程HSP中的obfuscation.txt文件。</li>     </ul>     <p>如果构建HAR，生成的远程HAR中的obfuscation.txt是以下文件的合并：</p>     <ul>      <li>自身的consumerFiles属性。</li>      <li>依赖的本地HSP的consumerFiles属性。</li>      <li>依赖的本地HAR的consumerFiles属性。</li>      <li>依赖的远程HAR和远程HSP中的obfuscation.txt文件。</li>     </ul>     <p>构建HSP时，生成的远程HSP中的obfuscation.txt仅包含自身的consumerFiles属性。</p>     <p>构建HAP时，不会生成obfuscation.txt文件。</p>     <p><strong>混淆规则合并逻辑</strong></p>     <p>混淆选项：使用或运算进行合并，即开关选项只要在参与合并的任意一个规则文件中存在，最终的合并结果中就会包含该开关选项。</p>     <p>保留选项：合并时，对于白名单选项，其内容取并集。</p>     <ul>      <li>       <p><strong>如果当前编译模块混淆配置未包含-enable-lib-obfuscation-options选项</strong>：合并对象为当前模块的所有混淆规则与依赖模块混淆规则中的<a href="/consumer/cn/doc/harmonyos-guides/source-obfuscation#保留选项">保留选项</a>。</p></li>      <li>       <p><strong>如果当前编译模块混淆配置包含-enable-lib-obfuscation-options选项</strong>：合并对象为当前模块的所有混淆规则与依赖模块的所有混淆规则。</p></li>     </ul>     <p>对于API version 18之前版本，如果consumerFiles指定的混淆配置文件中包含以下混淆选项和保留选项，这些规则将被合并到远程HAR和HSP的obfuscation.txt文件中，其他混淆规则不会被合并。</p>     <div _ngcontent-oqj-c106="" class="highlight-div"><div _ngcontent-oqj-c106=""
